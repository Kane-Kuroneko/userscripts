# 更新日志与 Readme 编写规范

## 核心原则

CHANGELOG.md 和 Readme.md 是面向**脚本使用者**的文档，不是给开发者看的技术笔记。

每个版本条目必须从**用户视角**描述变化：用户能感知到什么、操作上有什么不同。禁止写入实现细节、技术方案、防御策略、架构决策等内容。

## 对比示例

| ❌ 错误写法（技术细节） | ✅ 正确写法（用户视角） |
|-------------------------|------------------------|
| 适配 acgxj su-download 新版 DOM（button[data-qr-url] base64 直链） | 适配 acgxj.com 新版下载按钮，二维码转直链功能恢复正常 |
| 修复 steamzg 等页面因 ResizeObserver 循环触发 webpack HMR overlay 红色错误遮罩；三层防御：capture-phase 拦截 + window.addEventListener 劫持 + CSS overlay 隐藏 | 修复 steamzg.com 等页面偶现红色错误遮罩遮挡内容的问题 |
| 修正 7.2.0 发布文档错误：恢复被误删的 7.1.0 补丁条目 | 改进更新日志排版，增强可读性 |
| 适配 gamer520 新版下载页 .bdp-container 卡片式布局 | 适配 gamer520 新版下载页布局，二维码自动转直链 |

## 禁止出现的内容

- 选择器名、DOM 结构名、类名、ID
- 技术方案、防御策略、实现层次（如"三层防御"）
- 内部文件名、函数名、模块名
- 发布流程元信息（如"恢复被误删的条目""修正 Readme.md 文档区域"）
- 版本号争议、发布事故、agent 操作记录

## 允许保留的用户视角描述

- "修复 XX 问题" — 描述用户遇到的现象，不解释原因
- "新增 XX 功能" — 描述用户能用的新能力
- "适配 XX 站点新版页面" — 说明站点变化
- "增强可读性/优化体验" — 描述用户体验改善

## 例外（留存技术细节的场景）

技术细节只允许出现在 `docs/` 目录下的分析文档中（如 `docs/resizeobserver-loop-error.md`），供开发者排查问题时参考。
