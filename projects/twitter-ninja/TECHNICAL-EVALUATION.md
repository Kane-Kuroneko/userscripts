## 项目概述

### 项目定位
Twitter Ninja 是一款专注于提升 Twitter 电脑端用户体验的浏览器扩展/Userscript，旨在解决用户使用 Twitter 时的核心痛点。

### 核心功能
1. **[super block]** - 云端共享黑名单系统 (付费功能)
2. **[fast browse]** - 快速浏览与树状评论
3. **[auto block]** - 自动屏蔽机制

---

## 功能需求分析

### 1. [super block] 云共享黑名单

#### 功能描述
- 用户点击 [super block] 按钮即可上传用户 ID 至云端共享名单
- 刷推时自动匹配云端名单，对评论区用户执行 [original block] 及 [super block]
- 付费订阅模式: $1/月，支持 1-2 周试用

#### 用户价值
- **社区驱动**: 集体智慧，共同维护高质量浏览环境
- **自动化**: 无需手动管理大量屏蔽列表
- **实时性**: 云端同步，即时生效

#### 技术需求
- 用户认证系统 (注册/登录)
- 云端数据库 (存储黑名单)
- 付费系统集成 (支付/订阅/试用管理)
- 实时同步机制
- 数据隐私保护

---

### 2. [fast browse] 快速浏览

#### 功能描述
- 点击推文后弹窗打开内容，而非跳转路由
- 支持以 Reply 形式展示树状评论
- 清晰可见回复的具体评论

#### 用户价值
- **效率提升**: 避免页面跳转，保持浏览上下文
- **体验优化**: 树状结构清晰展示对话脉络
- **沉浸式浏览**: 不丢失当前 Feed 位置

#### 技术需求
- DOM 事件拦截 (阻止默认路由跳转)
- 弹窗组件开发
- Twitter 内部 API 数据抓取
- 树状评论渲染逻辑
- 性能优化 (避免内存泄漏)

---

### 3. [auto block] 自动屏蔽

#### 功能描述
- 检测评论区中屏蔽了自己的用户
- 自动执行 super block 该用户

#### 用户价值
- **零操作**: 全自动处理，无需手动干预
- **情绪保护**: 避免看到敌对用户的评论
- **互惠机制**: "你屏蔽我，我也屏蔽你"

#### 技术需求
- 用户关系检测 (判断是否被屏蔽)
- 后台定期扫描机制
- 请求频率控制 (避免触发反爬)
- 自动化屏蔽逻辑

## 技术难度评估

### 难度等级说明
- ⭐ (1星): 简单 - 常规前端开发，1-3天
- ⭐⭐ (2星): 较简单 - 需一定经验，3-7天
- ⭐⭐⭐ (3星): 中等 - 需要专项技术，1-2周
- ⭐⭐⭐⭐ (4星): 中高 - 复杂逻辑+风险控制，2-4周
- ⭐⭐⭐⭐⭐ (5星): 高 - 全栈开发+商业系统，4-6周+

---

### [super block] - ⭐⭐⭐⭐⭐ (高难度)

| 维度 | 评分 | 说明 |
|------|------|------|
| 前端开发 | ⭐⭐⭐ | 弹窗UI、设置面板、状态管理 |
| 后端开发 | ⭐⭐⭐⭐⭐ | 用户系统、数据库、API、支付集成 |
| 数据安全 | ⭐⭐⭐⭐⭐ | 隐私保护、GDPR合规、数据加密 |
| 运维部署 | ⭐⭐⭐⭐ | 服务器维护、监控、备份 |
| 测试验证 | ⭐⭐⭐⭐ | 多场景测试、支付流程测试 |

**技术难点**:
1. **Twitter 无官方 API**: 需要逆向工程内部接口，随时可能失效
2. **付费系统**: 需要集成 Stripe/Paddle，处理订阅、退款、试用逻辑
3. **实时同步**: 需要 WebSocket 或长轮询，保证名单实时更新
4. **用户认证**: JWT/OAuth2，安全存储密码
5. **数据规模**: 用户量大时需要数据库优化和缓存策略

**核心挑战**:
- 构建完整的 SaaS 系统 (不仅仅是浏览器插件)
- 持续的服务器运维成本
- 支付合规性 (不同国家法规)

---

### [fast browse] - ⭐⭐⭐ (中等难度)

| 维度 | 评分 | 说明 |
|------|------|------|
| DOM 操作 | ⭐⭐⭐⭐ | 事件拦截、路由劫持、动态渲染 |
| API 逆向 | ⭐⭐⭐⭐ | 抓取 Twitter 内部 API |
| UI 开发 | ⭐⭐⭐ | 弹窗、树状评论组件 |
| 性能优化 | ⭐⭐⭐ | 内存管理、虚拟滚动 |
| 兼容性 | ⭐⭐⭐ | 适配 Twitter UI 更新 |

**技术难点**:
1. **路由拦截**: 需要拦截 Twitter SPA 的路由跳转 (History API)
2. **数据获取**: 解析 GraphQL 请求，提取推文和评论数据
3. **树状渲染**: 递归渲染回复关系，处理深度嵌套
4. **弹窗定位**: 适配不同屏幕尺寸和滚动位置

**核心挑战**:
- Twitter 频繁更新 DOM 结构，需要持续适配
- 大量评论时的性能优化 (虚拟列表)
- 避免与 Twitter 原生脚本冲突

---

### [auto block] - ⭐⭐⭐⭐ (中高难度)

| 维度 | 评分 | 说明 |
|------|------|------|
| 关系检测 | ⭐⭐⭐⭐⭐ | Twitter 不直接提供"谁屏蔽了我"的接口 |
| 后台运行 | ⭐⭐⭐ | 定时任务、队列管理 |
| 频率控制 | ⭐⭐⭐⭐ | 避免触发反爬机制 |
| 容错处理 | ⭐⭐⭐⭐ | 网络异常、API 变更处理 |

**技术难点**:
1. **屏蔽检测**: Twitter 不公开此信息，需要间接推断
   - 方案 A: 尝试关注对方，失败则可能被屏蔽
   - 方案 B: 分析互动数据 (评论可见性)
   - 方案 C: 调用内部 API (风险高)
2. **频率限制**: Twitter 对 API 请求有严格限制
   - 需要实现请求队列和退避策略
   - 随机化请求间隔
3. **误判风险**: 可能导致错误屏蔽

**核心挑战**:
- 如何准确检测"被屏蔽"状态
- 平衡自动化与安全性
- 账号封禁风险控制

## 技术架构方案

### 技术选型对比

| 维度 | Userscript | Chrome Extension |
|------|-----------|------------------|
| 开发难度 | ⭐⭐ | ⭐⭐⭐ |
| 跨平台性 | ✅ 优秀 (支持多种浏览器) | ❌ 仅 Chromium 系 |
| UI 能力 | ⭐⭐ (受限) | ⭐⭐⭐⭐⭐ (完整) |
| 存储能力 | ⭐⭐ (GM API) | ⭐⭐⭐⭐ (chrome.storage) |
| 后台运行 | ❌ 依赖页面 | ✅ Service Worker |
| 审核门槛 | ⭐ 无 | ⭐⭐⭐ Chrome Web Store |
| 商业化 | ⭐⭐ 困难 | ⭐⭐⭐⭐ 容易 |
| 安装门槛 | ⭐ 需 Tampermonkey | ⭐⭐⭐ 一键安装 |

### 推荐方案

**MVP 阶段**: Userscript
- 快速迭代验证需求
- 跨平台通用
- 安装门槛低

**商业化阶段**: Chrome Extension
- 独立 UI 界面
- 付费验证更安全
- 后台同步更稳定
- 用户安装更便捷

## 关键决策点

### Decision 1: Userscript vs Extension
**建议**: MVP 用 Userscript，商业化转 Extension

**理由**:
- 快速验证需求 (Userscript)
- 降低初期开发成本
- 商业化时需要更强 UI 和后台能力 (Extension)

---

### Decision 2: 付费功能优先级
**建议**: 先做免费功能积累用户，再推出付费功能

**理由**:
- 降低用户使用门槛
- 验证功能价值后再收费
- 避免初期用户流失

---

### Decision 3: 后端技术选型
**建议**: Node.js + PostgreSQL + Stripe

**理由**:
- 前端 TypeScript 技术栈一致
- PostgreSQL 成熟稳定
- Stripe 开发者体验最佳
